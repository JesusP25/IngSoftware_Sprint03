Demostrar y validar el sprint

Gestión de Software 27 de abril, 2025

Es el proceso continuo de revisar, detallar y reordenar las historias del Product Backlog. Se dividen tareas grandes y se ajustan prioridades. Esto asegura que el equipo siempre tenga historias bien definidas y listas para futuros sprints.

Entregables del Sprint

Historia de Usuario Entregables
Historia de Usuario 1: Trámites Archivados
  • Como usuario del sistema, quiero poder buscar un trámite archivado ingresando su número de expediente para visualizar su información detallada de manera rápida y organizada.
  • Funcionalidad de búsqueda por número de expediente.
  • Tabla de resultados con campos clave del trámite.
  • Mensajes de retroalimentación en caso de error.
  • Interfaz optimizada para escritorio y dispositivos móviles.
Historia de Usuario 2: Trámites Derivados
  • Como usuario del sistema, quiero consultar trámites derivados utilizando el mismo formulario de búsqueda para gestionar eficazmente los diferentes tipos de trámites desde un solo lugar.
  • Módulo de búsqueda de trámites derivados con campos reutilizables.
  • Tabla detallada de resultados.
  • Compatibilidad responsive.
  • Mensajes en caso de búsqueda vacía o error.
Historia de Usuario 3: Seguimiento de Trámite
  • Como usuario del sistema, quiero visualizar el seguimiento completo de un trámite con todos sus pasos registrados y estado actual de manera clara.
  • Tabla de historial del trámite con columnas: fecha, acción, y área.
  • Ordenamiento cronológico automático.
  • Diseño adaptable a pantalla móvil.
  • Componente de mensaje vacío.
Historia de Usuario 4: Listado de remitentes
  • Como usuario administrativo, quiero buscar, editar y eliminar remitentes desde un listado con filtros personalizados para mantener actualizada la base de datos de contactos.
  • Se puede buscar remitentes por nombre, celular o ID.
  • Cada remitente tiene opciones de “Editar” y “Eliminar”.
  • Las ediciones se guardan correctamente y se reflejan en el listado.
  • Se pide confirmación antes de eliminar un remitente.
  • El listado es ordenado, responsivo y muestra datos clave (nombre, celular, correo).
Historia de Usuario 5: Gestión de administradores
  • Como usuario administrador, quiero gestionar la lista de administradores con funciones de edición y eliminación para mantener la seguridad del sistema.
  • Se presenta un listado de todos los administradores del sistema con sus datos relevantes.
  • Cada administrador puede ser editado con validación de campos obligatorios.
  • Las modificaciones se reflejan de inmediato en el listado.

Criterios determinados

  • Consulta de Trámites Archivados y Derivados:
    • Búsqueda funcional por número de expediente, fechas y asunto.
    • Resultados visibles en tablas bien estructuradas.
    • Validaciones completadas para campos vacíos o mal ingresados.
    • Mensajes de error claros y funcionalidad responsive implementada.
  • Seguimiento de Trámites:
    • Visualización completa del historial del trámite (con área, fecha y acción).
    • Diseño ordenado y responsivo.
    • Datos cargados de manera rápida y sin errores.
  • Gestión de Remitentes:
    • Listado funcional con filtros por nombre, celular o ID.
    • Funcionalidad para editar y eliminar registros con confirmaciones activas.
    • Cambios reflejados inmediatamente.
    • Pruebas exitosas en dispositivos móviles y escritorio.
  • Gestión de Administradores:
    • Listado de administradores con funciones de edición y eliminación.
    • Validación completa de campos al editar.
    • Seguridad activa: no se permite eliminar al superadministrador.
    • Interfaz usable y consistente.
  • Pruebas de Calidad (QA):
    • Validación de funcionamiento en distintos navegadores y dispositivos.
    • Verificación de rendimiento y carga sin errores críticos.
    • Validación de todos los módulos del sprint.
  • Aprobación de Stakeholders:
    • Validación del entregable con feedback positivo.
    • Ajustes realizados según observaciones del equipo o cliente.
    • Los entregables son presentados a los stakeholders clave (institución, secretarios técnicos, jueces, etc.) para su validación.
    • Se recopila retroalimentación formal sobre la funcionalidad, usabilidad y cumplimiento de requisitos.
    • Se implementan los ajustes necesarios en base a las observaciones recibidas, asegurando que el producto cumpla con las expectativas antes del cierre del sprint.

Criterios de Aceptación

Historia de Usuario Criterios de Aceptación
Historia de Usuario 1
  • El usuario puede ingresar un número de expediente y obtener resultados detallados.
  • Se muestra una tabla clara con los datos del trámite (tipo, asunto, fecha, etc.).
  • Si no se encuentra ningún trámite, se muestra un mensaje informativo.
  • El campo de búsqueda debe validar que no esté vacío y que tenga al menos 3 caracteres.
  • El diseño es compatible con diferentes dispositivos.
Historia de Usuario 2
  • El formulario de búsqueda funciona igual para trámites derivados y archivados.
  • La tabla de resultados muestra tipo, asunto, estado y fecha.
  • Se validan los rangos de fechas y los campos ingresados.
Historia de Usuario 3
  • Al seleccionar un trámite, se muestra su historial completo en una tabla con campos como: acción, fecha, área, y observaciones.
  • La tabla está ordenada cronológicamente de manera descendente.
  • Si no hay historial, se muestra una notificación adecuada (“Este trámite no tiene seguimiento registrado”).
  • Interfaz compatible con móviles y escritorio.
Historia de Usuario 4
  • Se pueden buscar remitentes por nombre, celular o ID con filtros visibles.
  • Las opciones de editar y eliminar funcionan correctamente.
  • La eliminación requiere confirmación.
  • Los cambios en la tabla se reflejan inmediatamente.
Historia de Usuario 5
  • El listado muestra los datos relevantes de cada administrador.
  • La edición permite modificar nombre, correo y permisos con validación.
  • La eliminación pide confirmación.
  • Toda la interfaz es accesible y responsiva.
