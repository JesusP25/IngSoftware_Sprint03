Identificar tareas

Gestión de Software 27 de abril, 2025

La identificación de tareas es importante porque al momento de que el equipo se compromete a completar ciertas historias durante un sprint, es necesario dividirlas en tareas más pequeñas para facilitar la planificación, la asignación de responsabilidades y el seguimiento del progreso. Esta identificación permite al equipo trabajar de manera organizada y eficiente, asegurando que todos los miembros sepan exactamente qué deben hacer para alcanzar los objetivos del sprint y entregar valor al cliente dentro del tiempo establecido.

Historias de usuario comprendidas

Historia de usuario 1: Como usuario del sistema, quiero poder buscar un trámite archivado ingresando su número de expediente para visualizar su información detallada de manera rápida y organizada.

Historia de usuario 2: Como usuario del sistema, quiero consultar trámites derivados utilizando el mismo formulario de búsqueda para gestionar eficazmente los diferentes tipos de trámites desde un solo lugar.

Historia de usuario 3: Como usuario del sistema, quiero visualizar el seguimiento completo de un trámite con todos sus pasos registrados para entender su evolución y estado actual de manera clara.

Historia de usuario 4: Como usuario administrativo, quiero buscar, editar y eliminar remitentes desde un listado con filtros personalizados para mantener actualizada la base de datos de contactos.

Historia de usuario 5: Como usuario administrador, quiero gestionar la lista de administradores con funciones de edición y eliminación para mantener el control y la seguridad del sistema en tiempo real.

Herramientas

Reuniones de planificación de Sprint

En esta reunión, el equipo Scrum, junto con el Product Owner, define qué historias de usuario o elementos del backlog se priorizan y se comprometen a completarlas dentro del sprint. El equipo también desglosa estas historias en tareas más pequeñas, asigna responsabilidades y estima el esfuerzo necesario.

Salidas

Lista de tareas

Como resultado de la planificación, se genera una lista detallada de tareas comprometidas, con responsables asignados y fechas estimadas para su entrega dentro del sprint.

Historia 1:

Como usuario del sistema, quiero poder buscar un trámite archivado ingresando su número de expediente para visualizar su información detallada de manera rápida y organizada.

  • Tarea 1.1: Diseñar el formulario de búsqueda con campo para número de expediente y botón “Buscar”.
  • Tarea 1.2: Validar el formato del número de expediente antes de realizar la búsqueda.
  • Tarea 1.3: Implementar la lógica frontend para enviar el número al backend mediante una solicitud HTTP.
  • Tarea 1.4: Configurar el endpoint en el backend que reciba el número de expediente y devuelva los datos del trámite archivado.
  • Tarea 1.5: Mostrar los resultados en una tabla paginada con formato legible.
  • Tarea 1.6: Manejar mensajes de error si no se encuentra el expediente o si hay problemas de conexión.

Historia 2:

Como usuario del sistema, quiero consultar trámites derivados utilizando el mismo formulario de búsqueda para gestionar eficazmente los diferentes tipos de trámites desde un solo lugar.

  • Tarea 2.1: Reutilizar el formulario de búsqueda ya existente para incluir la opción de buscar trámites derivados.
  • Tarea 2.2: Implementar en el backend la lógica que identifique y devuelva trámites derivados según los criterios definidos.
  • Tarea 2.3: Ajustar la respuesta del backend para que distinga trámites derivados de otros tipos.
  • Tarea 2.4: Mostrar los trámites derivados en una tabla paginada con columnas relevantes.
  • Tarea 2.5: Agregar mensajes informativos si no se encuentran trámites derivados con el número ingresado.

Historia 3:

Como usuario del sistema, quiero visualizar el seguimiento completo de un trámite con todos sus pasos registrados para entender su estado actual de manera clara.

  • Tarea 3.1: Diseñar la sección de “Seguimiento” con una tabla que detalle cada paso del trámite.
  • Tarea 3.2: Definir y estructurar los campos necesarios: código, tipo de documento, asunto, fecha de registro e historial de trámite.
  • Tarea 3.3: Crear el endpoint del backend para consultar el historial completo de un trámite por su ID o código.
  • Tarea 3.4: Implementar la lógica frontend para consumir el historial del trámite y mostrarlo ordenadamente.

Historia 4:

Como usuario administrativo, quiero buscar, editar y eliminar remitentes desde un listado, para así mantener actualizada la base de datos de contactos.

  • Tarea 4.1: Diseñar el listado de remitentes con una interfaz amigable y botones visibles para editar y eliminar.
  • Tarea 4.2: Implementar filtros por nombre, celular e ID con campos de búsqueda dinámicos.
  • Tarea 4.3: Configurar el backend para responder a búsquedas filtradas y devolver los datos adecuados.
  • Tarea 4.4: Configurar la funcionalidad de eliminación con confirmación previa antes de borrar un remitente.
  • Tarea 4.5: Mostrar mensajes de validación si hay campos vacíos o datos mal ingresados.

Historia 5:

Como usuario administrador, quiero gestionar la lista de administradores con funciones de edición y eliminación para mantener el control y la seguridad del sistema.

  • Tarea 5.1: Diseñar el listado de administradores en una tabla clara con opciones de editar y eliminar.
  • Tarea 5.2: Implementar una alerta de confirmación antes de eliminar un administrador.
  • Tarea 5.3: Asegurar que todos los cambios se reflejen en la tabla.
  • Tarea 5.4: Validar permisos para que solo usuarios autorizados puedan editar o eliminar administradores.
