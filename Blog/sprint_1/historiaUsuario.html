Crear historias de usuario

Gestión de Software 27 de abril, 2025

Las historias de usuario son descripciones simples y concisas de funcionalidades que deben desarrollarse desde la perspectiva del usuario, facilitando la comunicación entre los interesados y el equipo técnico. Estas historias ayudan a dividir el trabajo en pequeños incrementos manejables, que luego pueden priorizarse, planificarse y completarse dentro de cada sprint, asegurando una entrega constante de valor y un enfoque en los objetivos más importantes del proyecto.

Criterios terminados

Épica 1: Trámites archivados

  • El formulario permite ingresar el número de expediente.
  • Al hacer clic en "Buscar", se realiza una consulta al backend y se muestra el resultado en una tabla.
  • Los resultados se presentan paginados para facilitar la lectura.

Épica 2: Trámites derivados

  • El mismo formulario de búsqueda funciona para trámites derivados.
  • El backend devuelve correctamente los trámites derivados.
  • Los resultados también son presentados en tabla paginada.

Épica 3: Seguimiento de trámites

  • Toda la información debe cargarse de manera clara y ordenada.
  • Se incluye una tabla que registra cada paso del trámite.
  • Muestra los datos (código, tipo de documento, asunto, fecha de registro, historia de trámite).

Épica 4: Listado de remitentes

  • Se muestra un buscador con filtros por nombre, celular e ID.
  • Se habilitan botones para editar y eliminar cada remitente.
  • Se muestra un mensaje cuando hay campos vacíos.

Épica 5: Listado de administradores

  • Cada registro puede editarse desde la lista con un botón dedicado.
  • La eliminación muestra una alerta de confirmación.
  • Todos los cambios deben reflejarse en la tabla en tiempo real.

Salidas

Historias de usuario

Historia de usuario 1: Como usuario, quiero ingresar el número de expediente en un formulario para poder buscar un trámite archivado específico.

Historia de usuario 2: Como usuario, quiero visualizar los resultados de búsqueda en una tabla paginada para leerlos con mayor facilidad.

Historia de usuario 3: Como usuario, quiero usar el mismo formulario de búsqueda para consultar trámites derivados sin tener que llenar datos adicionales.

Historia de usuario 4: Como sistema, quiero consultar al backend los trámites derivados para asegurar que los resultados mostrados sean precisos.

Historia de usuario 5: Como usuario, quiero ver los trámites derivados en una tabla paginada para revisarlos de manera ordenada.

Historia de usuario 6: Como usuario, quiero que toda la información del seguimiento de trámites se presente de forma clara para poder entender fácilmente el proceso.

Historia de usuario 7: Como usuario, quiero acceder a una tabla que registre cada paso del trámite para conocer su historial completo.

Historia de usuario 8: Como usuario, quiero ver los datos del trámite (código, tipo, asunto, fecha de registro e historial) para tener una visión detallada del proceso.

Historia de usuario 9: Como usuario, quiero contar con un buscador que filtre remitentes por nombre, celular e ID para encontrar rápidamente a la persona que necesito.

Historia de usuario 10: Como usuario, quiero recibir un mensaje cuando haya campos vacíos para evitar errores al buscar o editar remitentes.

Historia de usuario 11: Como usuario, quiero poder editar los datos de un administrador directamente desde la lista para actualizar la información rápidamente.

Historia de usuario 12: Como usuario, quiero ver una alerta de confirmación antes de eliminar un administrador para evitar borrados accidentales.

Criterios de aceptación de las historias

Historia Criterios de Aceptación
Historia 1
  • El formulario debe permitir ingresar únicamente valores válidos como número de expediente.
  • Al hacer clic en "Buscar", se debe consultar al backend y devolver datos correctos relacionados al expediente ingresado.
  • Los resultados deben mostrarse en una tabla ordenada con paginación.
  • Si no se encuentra información, debe mostrarse un mensaje claro indicando que no hay resultados.
  • La tabla debe permitir navegar entre páginas de resultados sin errores.
Historia 2
  • El formulario de búsqueda debe funcionar también para trámites derivados sin requerir cambios visuales.
  • La respuesta del backend debe incluir únicamente trámites clasificados como derivados.
  • Los resultados deben mostrarse en una tabla paginada con los campos correspondientes.
  • Se debe validar que la tabla paginada cargue correctamente al cambiar de página.
  • Si no hay trámites derivados para mostrar, debe aparecer un mensaje indicando "Sin resultados".
Historia 3
  • El sistema debe mostrar toda la información del trámite de forma estructurada y clara.
  • La tabla debe registrar y visualizar cada paso o acción relacionada al trámite.
  • Se deben mostrar los campos: código, tipo de documento, asunto, fecha de registro e historial del trámite.
  • La información debe actualizarse correctamente si hay nuevas acciones en el historial.
  • El usuario debe poder distinguir cada paso del trámite sin confusiones.
Historia 4
  • El buscador debe permitir filtrar por nombre, celular e ID con resultados precisos.
  • Los botones de editar y eliminar deben estar visibles y funcionales para cada remitente.
  • Al intentar ejecutar acciones con campos vacíos, el sistema debe mostrar mensajes de validación.
  • Las acciones de edición y eliminación deben reflejarse inmediatamente en la tabla.
  • Si no hay remitentes que cumplan los filtros, debe mostrarse un mensaje adecuado.
Historia 5
  • Cada registro debe tener un botón de edición que permita modificar los datos sin salir de la lista.
  • Al hacer clic en "Eliminar", debe mostrarse una alerta de confirmación antes de proceder.
  • Si el usuario confirma la eliminación, el registro debe eliminarse y desaparecer de la tabla en tiempo real.
  • Si el usuario cancela la eliminación, no debe producirse ningún cambio.
  • Cualquier cambio en los datos debe reflejarse instantáneamente sin necesidad de recargar la página.
